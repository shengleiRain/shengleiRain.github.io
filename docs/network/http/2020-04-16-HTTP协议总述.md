## 概述

HTTP全称为Hypertext Transfer Protocol（超文本传输协议），是一种用于**分布式、协作式和超媒体信息系统**的应用层协议。同时，它也是一种**无状态**的协议，通过扩展**请求方法、错误码、请求头**来用于实现不仅限于超文本的许多任务，例如**名称服务器和分布式对象管理系统。**

> 所谓无状态是指在多次请求中，协议**不会要求HTTP服务器保留用户的信息或状态**。也就是说，如果后面的请求需要之前的用户信息或状态，就必须重新请求重传。但是，web应用实现为了保持HTTP连接状态信息，通常会采用**Cookie或者Session技术**。

HTTP协议是为了实现**客户端和服务器之间的请求和应答**而制定的标准，通常使用**TCP/IP协议簇**。但是，HTTP协议并**没有规定**应用层以下的层级所必须要使用的协议类型，只要**下层协议能够提供可靠连接服务**即可。例如 [HTTPU](https://en.wikipedia.org/wiki/HTTPU) 和 [Simple Service Discovery Protocol](https://en.wikipedia.org/wiki/Simple_Service_Discovery_Protocol) (SSDP)中的HTTP协议实现就是使用的**UDP协议**。**HTTP/3使用QUIC来提供传输层的连接传输服务，QUIC是一种基于UDP协议的传输层协议。**

不过**HTTP/1.1和HTTP/2**都是采用**TCP协议**作为传输层协议，以浏览器和服务器之间的请求响应为例。

- 浏览器向服务器发起一个HTTP请求，该请求是由**统一资源标识符（URI）**确定，其**scheme为http或者https**，示例如图：

  ![截屏2020-04-16下午12.18.36](../../assets/images/git/%E6%88%AA%E5%B1%8F2020-04-16%E4%B8%8B%E5%8D%8812.18.36.png)

- 浏览器和服务器之间经过TCP三次握手建立起TCP连接

- 服务器向浏览器返回请求响应，响应浏览器所请求的资源，例如HTML文件等

- 浏览器解析HTML文件，进行网页渲染

## HTTP的发展历史以及版本对比

| 版本     | 年份 | 主要功能                                                     |
| -------- | ---- | ------------------------------------------------------------ |
| HTTP/0.9 | 1991 | **已过时**。首次将HTTP协议写成规范的文档，只接受**GET**一种请求方法，**没有在通讯中指定版本号**，且**不支持请求头**。 |
| HTTP/1.0 | 1996 | 第一个**在通讯中指定版本号**的HTTP协议版本。<br>- 支持**GET、HEAD和POST**这三种请求方法<br>- 添加一些**请求头字段**<br>- 不支持keep-alive机制，即**限制每次连接只能处理一个请求，服务器处理完用户的请求，发出响应之后，然后收到用户的应答后，就断开连接。** |
| HTTP/1.1 | 1999 | 版本规范在 [RFC2616](https://tools.ietf.org/html/rfc2616)<br>- **支持keep-alive机制**，多个请求能够复用同一个连接，避免了多次TCP三次握手过程，而且连接也得益于TCP协议的**慢启动机制**变得更快<br>- 支持以**管道方式**在同时发送多个请求，以便降低线路负载，提高传输速度<br>- 添加了五个新的请求方法：**OPTIONS、PUT、DELETE、TRACE和CONNECT** |
| HTTP/2.0 | 2015 | 版本规范在[RFC7540](https://tools.ietf.org/html/rfc7540)<br>- 兼容HTTP/1.1，例如**请求方法、请求头等**<br>- 考虑从以下几方面降低页面加载的时延：<br>.          - **对HTTP的请求头进行数据压缩**<br>.          - **HTTP/2服务器主动推送** <br>.          - **使用管道方式同时进行多个请求**<br>.          -  **解决HTTP/1.1中的Head-of-line  blocking问题，但是也无法完全解决**<br>.          - **在一个TCP连接上多路复用多个请求** |
| HTTP/3.0 | 2018 | **使用QUIC作为传输层协议**                                   |

> **Head-of-line blocking**是当前面序号的数据包无法收到时，后面数据包都被阻塞而造成的一种性能限制现象。这是由于**TCP协议**的接收到的数据包都需要**按序处理**，前面序号的数据包没收到，那么后面序号的数据包也不能处理。

## 请求消息格式

请求消息是由四部分组成：**请求行、请求头部、空行和请求数据（主体）**。

![img](../../assets/images/http/HTTP%E8%AF%B7%E6%B1%82%E6%B6%88%E6%81%AF%E7%BB%93%E6%9E%84.png)

### 请求行

请求行是由**请求方法、要访问的资源和所使用的HTTP版本**组成。

#### 请求方法

| 请求方法 | 作用                                                         | 安全性 | 幂等 | 可缓存 |
| -------- | :----------------------------------------------------------- | ------ | ---- | ------ |
| GET      | 向服务器请求资源数据，而且GET方法**应该只承担从服务器获取资源的任务**，不应该有其他的作用。 | 是     | 是   | 是     |
| HEAD     | 向服务器请求**响应头中的元信息**，响应中**没有主体数据**（response body） | 是     | 是   | 是     |
| POST     | 向服务器请求**接受**请求中的主体信息。                       | 否     | 否   | 是     |
| PUT      | 请求在服务器上**修改或创建**某些资源                         | 否     | 是   | 否     |
| DELETE   | 请求**删除**服务器上的特定资源                               | 否     | 是   | 否     |
| TRACE    | **对服务器接收到的请求进行回显**，以便客户端可以看到中间服务器所做的**更改或添加**的内容。 | 是     | 是   | 否     |
| OPTIONS  | 返回服务器所支持的指定URL的HTTP方法                          | 是     | 是   | 否     |
| CONNECT  | 将请求连接转换为TCP/IP隧道                                   | 否     | 否   | 否     |
| PATCH    | 对资源进行部分修改                                           | 否     | 否   | 否     |

> 所谓**安全方法**就是只向服务器请求数据，而不更改服务器中资源的状态。
>
> 所谓**幂等方法**是指多次相同的请求应该与单个请求具有相同的效果。

### 状态码

状态码在状态行中，用来显示请求的状态，分为5组：

- **1xx**：指示信息--表示请求已经接收，继续处理
- **2xx**：成功--表示请求已被成功接收
- **3xx**：重定向
- **4xx**：客户端错误--请求有语法错误或请求无法实现
- **5xx**：服务器错误
